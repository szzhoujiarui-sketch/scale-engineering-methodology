# Case Study: A 股个股分析场景的 Token 效率优化

Generated: 2026-05-19 during v10.1 token efficiency audit.

## 1. 场景描述

### 1.1 需求

用户向 Agent 发送指令：「分析一下贵州茅台（600519）今天的走势，结合最近的财报和新闻。」

Agent 需要从三个数据源获取信息并综合判断：

| 数据源 | 内容 | 访问方式 |
|--------|------|---------|
| K 线数据 | 日线行情、均线、成交量 | 通过 `/stock_quote` 工具调用实时数据 API |
| 财务数据 | 营收、净利润、毛利率、ROE | 通过 `/financial_report` 工具读取 |
| 新闻情绪 | 近期新闻标题、情绪打分 | 通过 `/news_sentiment` 工具抓取 |

### 1.2 初始技能结构

假设存在一个名为 `a-share-analysis` 的技能，其 `SKILL.md` 约 3200 汉字，references/ 目录下包含 4 个文件共约 5800 汉字。

```
a-share-analysis/
├── SKILL.md                      # 3200 汉字 ≈ 4800 tokens
├── references/
│   ├── stock-basics.md           # 1500 汉字
│   ├── financial-indicators.md   # 2000 汉字
│   ├── news-sentiment-method.md  # 1300 汉字
│   └── sample-analysis.md        # 1000 汉字（含完整分析示例）
```

---

## 2. 方法 1：触发词审计

### 2.1 审计步骤

列出 `a-share-analysis` 技能的 triggers 配置：

```yaml
triggers:
  - "股票"       # ← 问题：过于宽泛
  - "/分析"
  - "/财报"
  - "贵州茅台"
  - "/stock"
```

### 2.2 发现的问题

| 触发词 | 触发频率估算 | 误触率 | 判定 |
|--------|-------------|--------|------|
| `股票` | 高（普通对话中频繁出现） | ~80% | 🚫 无效触发 |
| `/分析` | 中 | ~40% | ⚠️ 偏宽泛 |
| `/财报` | 低 | ~10% | ✅ 可接受 |
| `贵州茅台` | 极低（唯一特定股票） | ~0% | ✅ OK，但实际场景应避免具体股票名 |
| `/stock` | 低 | ~5% | ✅ 可接受 |

**核心问题**：trigger `"股票"` 是一个日常高频中文词。用户在普通对话中提及「股票代码」「股票账户」「股票市场」时，都会触发加载整个技能，每次浪费约 4800 tokens（仅 SKILL.md）+ 参考资料（约 8700 tokens）= **约 13500 tokens/次**。

假设一天 50 次对话，其中 40 次是误触：

```
误触 token 浪费 = 40 × 13500 = 540,000 tokens/天
```

### 2.3 优化方案

将宽泛 trigger 替换为更精确的指令式触发词：

```yaml
triggers:
  - "/a股"        # 精确指令前缀
  - "/分析股票"    # 组合关键词，降低误触
  - "/财报分析"
```

**效果**：误触率从 ~80% 降至 ~5%，每天节省约 500,000 tokens。

## 3. 方法 2：技能内容瘦身

### 3.1 瘦身检查清单

逐项检查 `references/` 目录下的文件是否全部必要：

**□ references/stock-basics.md（1500 汉字）**
- 内容：K线图基本知识、均线类型、成交量含义
- 判定：Agent 通过工具调用 API 获取数据，这些基础知识在 Agent 的预训练知识中已存在
- 行动：🟡 **删除**，Agent 不需要在 prompt 中重新学习 K 线常识

**□ references/financial-indicators.md（2000 汉字）**
- 内容：毛利率、ROE、PE/PB 等指标的定义和计算公式
- 判定：Agent 的预训练模型已理解这些概念，无需额外注入
- 行动：🟡 **删除**，除非有特殊的 A 股专用指标差异需要说明

**□ references/news-sentiment-method.md（1300 汉字）**
- 内容：新闻情绪分析的方法论、典型打分逻辑
- 判定：这是 Agent 调用外部 API 的结果展示逻辑，API 返回已包含情绪分数
- 行动：🟡 **删除**，Agent 只需读取 API 返回的 score 字段

**□ references/sample-analysis.md（1000 汉字）**
- 内容：完整的分析示例，包含「假设用户问「茅台怎么样」，Agent 应该这样回复……」
- 判定：这是创建技能时用于测试的占位内容，不再是运行时必需
- 行动：🔴 **删除**，占位内容不应留在交付版本中

### 3.2 冗余来源分析

| 冗余类型 | 描述 | 占比估算 |
|---------|------|---------|
| **预训练重复** | LLM 已掌握的基础知识（K线、财务指标） | ~60% |
| **占位/测试内容** | 创建技能时复制的示例，从未实际使用 | ~20% |
| **API 逻辑重复** | 在 prompt 中描述 API 行为，而 Agent 应直接调用 API | ~15% |
| **用户偏好模板** | 用户可能用也可能不用的格式偏好 | ~5% |

### 3.3 优化后技能结构

```
a-share-analysis/
├── SKILL.md                      # 从 3200 汉字压缩至 1800 汉字
├── references/
│   └── a-share-apis.md           # 900 汉字（仅保留 API 文档摘要）
```

**节省统计**：

| 项目 | 优化前 | 优化后 | 节省 |
|------|-------|-------|------|
| SKILL.md | 3200 汉字 (~4800 tokens) | 1800 汉字 (~2700 tokens) | ~2100 tokens |
| references/ | 5800 汉字 (~8700 tokens) | 900 汉字 (~1350 tokens) | ~7350 tokens |
| **合计** | **~13500 tokens** | **~4050 tokens** | **~9450 tokens (70%)** |

## 4. 方法 3：模拟 token 成本估算

### 4.1 估算模型

采用附录 C 中的估算方法：

- SKILL.md 每 1000 汉字 ≈ 1500 tokens
- 每个 trigger 定义 ≈ 30 tokens
- 启动时基础开销：5 个技能 ≈ 2000 tokens

### 4.2 优化前后对比

#### 单次加载成本

| 项目 | 优化前 (tokens) | 优化后 (tokens) |
|------|----------------|----------------|
| SKILL.md 内容 | 4800 | 2700 |
| references/ 文件 | 8700 | 1350 |
| trigger 定义（5个 × 30） | 150 | 90（3个 × 30） |
| **单技能加载小计** | **13650** | **4140** |

#### 每轮对话总成本

假设本轮对话已加载了 5 个技能（含本技能）：

| 项目 | 优化前 (tokens) | 优化后 (tokens) |
|------|----------------|----------------|
| 5-skill baseline 开销 | 2000 | 2000 |
| a-share-analysis 加载 | 13650 | 4140 |
| 对话历史（均值） | 3000 | 3000 |
| 工具描述 | 1500 | 1500 |
| 系统提示 | 800 | 800 |
| **每轮对话总计** | **~20950** | **~11440** |

#### 月度成本推算

基于以下使用假设：
- 平均每天 50 轮对话
- 其中 30 轮涉及 A 股分析（含误触）
- 优化后误触率大幅降低，实际加载 5 轮/天

| 指标 | 优化前 | 优化后 |
|------|-------|-------|
| 日均加载次数（含误触） | 40 | 5 |
| 日均 token 消耗（仅本技能） | 40 × 13650 = 546,000 | 5 × 4140 = 20,700 |
| 月度 token 消耗（30 天） | ~16,380,000 | ~621,000 |
| 月度节省 | — | **~15,759,000 tokens** |

#### 按模型估算成本

假设使用 DeepSeek 或 GPT-4 级别模型（参考价 $3/M input tokens）：

| 项目 | 优化前 | 优化后 |
|------|-------|-------|
| 月度 input tokens | ~16.4M | ~0.6M |
| 月度成本（input 侧） | ~$49.2 | ~$1.9 |
| 年度成本 | ~$590 | ~$23 |

> 注：以上仅为 input token 成本。若考虑 output token 和推理开销，实际节省会更大。

## 5. 综合收益总结

### 5.1 三种方法的效果对比

| 方法 | 节省 tokens | 操作难度 | 见效速度 |
|------|------------|---------|---------|
| 触发词审计 | ~500,000 tokens/天（减少误触） | 低 | 立即 |
| 技能内容瘦身 | ~9,450 tokens/次（减少单次加载） | 中 | 永久 |
| 模拟成本估算 | —（提供量化依据，驱动决策） | 低 | 持续 |

### 5.2 经验教训

1. **Trigger 精确性是第一道防线**：宽泛 trigger 导致的误触是最大的 token 浪费来源，远超技能内容瘦身的节省量级
2. **不要相信「先写着，也许以后有用」**：每个 references/ 里的文件都应经过必要性检验，占位内容必须清理
3. **LLM 预训练知识 ≠ prompt 中重复的知识**：被删除的 2100 汉字的 K 线知识，LLM 自己都知道
4. **量化才能管理**：不做 token 成本估算，就不知道哪些优化优先级最高

### 5.3 优化后触发条件

完成审计后，建议将本案例的经验沉淀到 SCALE 方法论的核心技能中，作为 token 效率审计的标准参考。
