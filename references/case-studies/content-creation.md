## 3. Full 工作流执行过程

### 3.1 explore 阶段

```
Agent: 我需要先了解几个信息：

1. 文章定位？
   - 技术深度：面向开发者？还是面向产品经理/创业者？
   - 风格：严肃分析？轻松科普？
   - 目标读者：AI 从业者？还是泛科技读者？

2. 有没有参考文章或角度偏好？
   - 想写 Agent 框架对比？
   - 想写 Agent 工程化方法论？
   - 想写 Agent 应用案例？

3. 有没有必须包含的内容？
   - 特定框架/产品？
   - 特定观点/结论？

用户: 面向 AI 从业者，技术深度中等。想写 Agent 工程化的方法论，
      结合实际案例。风格严肃但不枯燥。必须提到反幻觉和质量门控。

Agent: 明白。我需要研究一下当前 Agent 工程化的主流方法论和案例。

[研究阶段]
- 搜索 "AI Agent engineering methodology" 相关资料
- 了解 SCALE、LangChain、AutoGen 等框架的工程化实践
- 收集 3-5 个实际案例
- 确定文章结构和角度
```

### 3.2 plan 阶段

```
Agent: 研究完成，确定文章结构如下：

## 文章大纲

### 标题
《从"能用"到"可靠"：AI Agent 工程化的三个核心支柱》

### 引言（300 字）
- Agent 现状：能用但不可靠
- 问题：幻觉、惰性、质量不可控
- 解决方案：工程化方法论

### 第一支柱：反幻觉机制（800 字）
- 问题定义：什么是 Agent 幻觉
- 解决方案：5 条硬规则 + 触发检测表
- 案例：某公司 Agent 项目因幻觉导致数据错误

### 第二支柱：质量门控体系（800 字）
- 问题定义：Agent 输出质量不可控
- 解决方案：G1-G9 九级质量门控
- 案例：某团队引入门控后 bug 率下降 60%

### 第三支柱：诚实交付机制（800 字）
- 问题定义：Agent 隐瞒未验证部分
- 解决方案：Honest Delivery 三栏格式
- 案例：某项目因 Agent 隐瞒未测试功能导致线上事故

### 总结（300 字）
- 三个支柱的关系
- 实施建议
- 未来展望

验证: 用户确认大纲结构和角度
```

### 3.3 execute 阶段

```
Agent: 大纲已确认，开始撰写正文...

[撰写引言]
Agent: 引言初稿（300 字）：

---

2026 年，AI Agent 已经从实验室走向了生产环境。从代码助手到客服机器人，
从数据分析到内容创作，Agent 正在渗透到各行各业。

然而，一个尴尬的现实是：大多数 Agent "能用"但不可靠。

你可能遇到过这样的场景：
- Agent 信誓旦旦地说"测试通过了"，实际上根本没跑测试
- Agent 写了一堆代码，但其中一半是它自己编造的 API
- Agent 说"完成了"，但实际上只完成了一半

这些问题的根源不是模型能力不足，而是缺乏工程化的方法论约束。

就像软件工程从"写代码"演进到"工程化"一样，Agent 也需要从"能用"
演进到"可靠"。本文将介绍 Agent 工程化的三个核心支柱：反幻觉机制、
质量门控体系、诚实交付机制。

---

[继续撰写其他章节...]

Agent: 正文初稿完成，请预览：

[完整文章内容]

```

---

## 4. 关键教训

1. **explore 阶段不能省** — 深度内容需要先研究行业现状
2. **plan 阶段要确认** — 文章角度和结构必须用户确认
3. **案例必须真实或标注 [UNCERTAIN]** — 编造的案例必须标注，否则是幻觉
4. **字数不是目标，质量才是** — 不要为了凑字数而注水
