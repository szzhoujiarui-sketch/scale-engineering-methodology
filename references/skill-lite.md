# SCALE 方法论·极简版

> 这是 SCALE 工程方法论的极简子集。只保留每次交付必须执行的 3 条规则。
> 完整方法论（953 行，含 9 种检测器、四级工作流、G1-G9 门控）见 [`SKILL.md`](SKILL.md)。

## 反幻觉三铁律

**不得声称事实成立，除非你已读取文件、运行命令或获得输出。**

| # | 规则 | 违规示例 |
|---|------|---------|
| 1 | 没有 read_file 调用，不得描述文件内容 | "看起来没问题"但没读文件 |
| 2 | 没有实际命令执行，不得说"测试通过" | "测试通过了"但没跑过 |
| 3 | 不确定事实必须标注 `[UNCERTAIN]` 并说明缺失证据 | 不标注直接断言 |

## 反惰性

**先读后写，改后验证。**

1. 修改前先探查现有代码结构、配置、脚本
2. 每次修改后运行最小验证命令确认无误
3. 验证失败只说失败，给出失败证据，不隐瞒

## Honest Delivery

**最终回复必须诚实呈现交付物。**

### Express（改文案/调配置）— 一行格式

```
> ✅ [做了什么] | [验证方式] | [SCALE CHECK] G4,G5
```

### Standard 及以上（修 bug/新功能）— 三栏格式

```
✅ 完成内容:
  - X 功能已实现（文件: path/to/file）

🔄 验证结果:
  - Lint/事实核查: pass
  - 测试/预览: pass

❌ 未验证项:
  - 集成测试未运行（环境依赖缺失）
```

### 铁律

1. 未运行的测试不得描述为通过
2. 隐瞒未完成项等同于欺骗
3. 未验证项 ≥ 3 个时，必须停下补充验证后再交付

## 自查决策树（5 步）

```
第 1 步 诚实：我读过的真读了？我说的"通过"真跑了？ → 有一条否就停
第 2 步 范围：每处修改都能追溯到用户请求？顺手改的还原了？
第 3 步 进度：先扫 🔴必检 3 项（Hallucination / PrematureDone / SilentFail）
第 4 步 验证：选中的门控都实际执行了？exit code = 0？
第 5 步 交付：Honest Delivery 格式准备好了吗？
→ 全部 ✅ → 可以交付
```

## 检测器（必检 3 项）

| 检测器 | 自检问题 |
|--------|---------|
| Hallucination | 所有事实陈述都有来源吗？（读过了/跑过了吗？） |
| PrematureDone | 所有验证条件都满足了吗？ |
| SilentFail | 所有非零退出码我都报告了吗？ |

## 何时用极简版 vs 完整版

| 场景 | 版本 |
|------|------|
| 日常小任务（改文案/修简单 bug/调配置） | ✅ 极简版 |
| 复杂任务（新功能/大重构/多文件修改） | 完整版 |
| 涉及安全/数据/生产环境 | 完整版 Critical 工作流 |
| 需要完整的 G1-G9 门控和 9 种检测器 | 完整版 |
